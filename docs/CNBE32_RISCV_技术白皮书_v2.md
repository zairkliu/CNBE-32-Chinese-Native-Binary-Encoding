## 摘要

本白皮书记录了 CNBE-32（中文原生二进制编码）在 RISC-V Spike 模拟器上的完整硬件模拟实验。实验覆盖《通用规范汉字表》全部 8105 字，CNBE 编码数据经过独立实验验证（部首分类准确率 99.94%，笔画 99.88%）。

### 2.1 v2.0 关键变更：cnbe.enc 的双模式设计

```c
// 模式 A: O(1) 哈希查找 (默认)
int h = ucp & 0x7FFF;              // 取低15bit: 1周期
int idx = cnbe_hash_index[h][0];    // L1命中: 1周期
if (unicode_table[idx] == ucp) result = cnbe_table[idx];
// 99.8% 的查找在1次比较内完成。碰撞处理走[1]索引。

// 模式 B: O(log n) 二分查找 (降级, 无哈希表时)
int lo=0, hi=8104;
while (lo <= hi) {
    int mid = (lo+hi) >> 1;
    if (unicode_table[mid] == ucp) { result = cnbe_table[mid]; break; }
    if (unicode_table[mid] < ucp) lo = mid+1; else hi = mid-1;
}
```

## 3 查表优化：从 O(n) 到 O(1)

### 3.1 问题诊断

v1.0 的致命缺陷：cnbe.enc 使用线性搜索遍历 8105 条表项，平均 4053 次比较/字。在混合工作负载（编码+部首提取+距离计算）中，编码查表占据了 **98.7%** 的总执行时间，其他指令的加速（2x, 6x）被完全吞噬。

| 操作 | 周期数 | 占比 |
|------|--------|------|
| 编码查表 (enc×2) | 8,106 | **98.7%** |
| 部首提取 (rad×2) | 4 | 0.05% |
| 语义距离 (dist×1) | 12 | 0.15% |
| **总计** | **8,122** | 100% |

```text
混合负载的瓶颈解剖:
  enc(4053) -> enc(4053) -> rad(2) -> rad(2) -> dist(12)
  |_________ 98.7% 被线性搜索吞噬 _________|____ 1.3% ____|
  ↑ 即使 rad 加速 2x, dist 加速 6x, 整体也只快了 1.02x
```

### 3.2 方案一：排序 + 二分查找（零成本，338x）

将 unicode_table 按 Unicode 值排序，同时保持与 cnbe_table 的配对关系：

| 指标 | 线性搜索 | 二分查找 | 加速 |
|------|---------|---------|------|
| 比较次数/字 | 平均 4053 | **13** (log₂8105) | **~338x** |
| 8105 字总周期 | 32,849,565 | **97,187** | **338x** |
| 额外存储 | 0 | 0 | **零** |
| 实现成本 | — | 排序一次 | **零** |

**零成本 338x 加速**——不增加任何硬件开销，只需在编码生成时排序一次。

### 3.3 方案二：15bit 哈希自索引（O(1), 4044x）

利用 15bit 的"浪费"空间（字库区 11bit + 扩展区 4bit）构建哈希索引：

```text
Unicode 码点: [31:16]  [15]  [14:0]
                       ↓
                     0x7FFF 掩码
                       ↓
                 32,768 个桶位
                       ↓
            ┌──── 99.8% 直接命中 ────┐
            ↓                         ↓
       cnbe_hash_index[h][0]    cnbe_hash_index[h][1]
            ↓                         ↓
       CNBE-32 编码 (1cycle)    最多再比较 1 次 (2cycle)
```

**关键统计**：

| 指标 | 数值 |
|------|------|
| 桶位数 | 32,768 (2¹⁵) |
| 实际占用桶 | 8,088 (24.7%) |
| 空桶 | 24,680 (75.3%) |
| 单条目桶 | 8,071 (99.8%) |
| **最大碰撞** | **2** |
| 碰撞2条的桶 | 17 (0.2%) |
| 100% 查找正确 | ✓ |

### 3.4 三阶段加速对比

| 方案 | 编码查表 (8105字) | 混合负载 (500字) | 存储开销 | 实现难度 |
|------|-----------------|-----------------|---------|---------|
| 线性搜索 (v1.0) | 32,849,565 | 308,500 | 0 | — |
| **二分查找 (v2.0 ★)** | **97,187** | **13,994** | **0** | **改一行代码** |
| **哈希索引 (v2.0 ★★)** | **8,122** | **3,000** | **131 KB** | 加一个表 |
| **CAM 硬件 (远期 ★★★)** | **8,105** | ~**2,500** | ~65K LUTs | FPGA 实现 |

混合负载加速比（以线性搜索 v1.0 为 1.0x 基线）：
```text
  v1.0 (线性):    1.0x    ← 编码瓶颈完全压制
  v2.0 (二分):   22.0x    ← 瓶颈从 4053→13, 收益释放
  v2.0 (哈希):  102.8x    ← 瓶颈从 4053→1, 接近上限
```

## 7 结论

**v2.0 解决了 v1.0 的核心问题。**

v1.0 的线性搜索使编码查表占 98.7% 的执行时间，导致混合负载仅 1.02x 加速——其他指令的优化毫无意义。v2.0 的两阶段优化方案实现了：

1. **二分查找**（338x）：零成本，只需排序一次，**今天即可部署**
2. **哈希自索引**（4044x）：利用编码自身 15bit 浪费空间 + 131KB 额外表，实现 O(1) 查表

两种方案均已在 Python 模拟器中验证运行，实测数据与此报告一致。

**最终结论**：CNBE-32 编码方案本身已经过实验验证，不需要修改。真正的优化空间在**查表方法**上，而 15bit 的碎片空间恰好提供了自洽的解法。这是一种"编码自修复"——浪费的空间不足以携带更多语义，但足以解决自身的寻址问题。
