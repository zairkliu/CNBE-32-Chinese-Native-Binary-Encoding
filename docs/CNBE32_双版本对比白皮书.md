# CNBE-32 双版本对比技术白皮书

**版本**: 3.0  
**日期**: 2026-07-01  
**对比版本**: 8105 字 (通用规范汉字表) vs 97,686 字 (全部 CJK 统一汉字)  

---

## 摘要

本白皮书对 CNBE-32 编码的两个版本进行系统性对比：**基础版**（覆盖《通用规范汉字表》8,105 字）和 **全量版**（覆盖 Unicode 17.0 CJK 全部统一汉字 97,686 字）。全量版在基础版基础上扩展了 12 倍字符数，修复了自动结构检测退化问题，将结构类型从 3 种恢复到 9 种。

两版共享相同的 32-bit 位域方案和 6 条 RISC-V 自定义指令，关键差异在于结构检测的自动化和索引溢出处理。

### 2.2 结构检测退化与修复

**基础版**：结构数据来自 CNBE V6.0 的手动标注，覆盖 6 种结构类型：
> 独体 + 左右 + 左中右 + 上下 + 上中下 + 品字 = **6 种**

**全量版（修复前）**：自动检测仅使用了 LEFT_RADICALS 和 TOP_RADICALS 两套规则，退化到 **3 种**：
> 左右 + 上下 + 上中下（默认） = **3 种 ↓**

**全量版（修复后）**：增加了 S2T 字典（19 个特例部首→精确结构类型映射），包括包围结构细分和独体结构识别，恢复到 **9 种**：
> 独体 + 左右 + 上下 + 左上包围 + 右上包围 + 左下包围 + 上三包围 + 下三包围 + 全包围 = **9 种 ↑**

**仍缺失的类型**：左中右 (2)、上中下 (4)、品字 (11) — 这 3 种需要字符级形态分析才能判断，无法通过部首编号自动识别。

## 三、编码空间利用率对比

### 3.1 位域利用

| 字段 | 宽度 | 基础版 | 全量版 | 说明 |
|------|:----:|:------:|:------:|------|
| 部首区 | 8bit | 213/256 (83.2%) | **214/256 (83.6%)** | 全量版补全了最后一个部首 |
| 笔画区 | 5bit | 28/32 (87.5%) | **31/32 (96.9%)** | 全量版覆盖所有笔画值 |
| **结构区** | **4bit** | **5/16 (31.2%)** | **9/16 (56.2%)** | **修复后 +4 种** |
| 字库区 | 11bit | 58/2048 (2.8%) | 组内 <2048 | 无溢出 |
| 扩展区 | 4bit | 0/16 (0%) | 0/16 (0%) | 两版均可用作元数据 |
| **有效信息** | **17bit** | **53.1%** | **56.2%** | +3.1pp |

### 3.2 扩展区释放

**修复后**两版的扩展区使用均为 **0 字 (0%)**。对于全量版，这是 9 种结构类型细分后(radical, stroke, structure) 每组均少于 2048 个成员的结果。

释放的 4bit 可用于：
```
扩展区 [3:0] 建议用途:
  [3] 繁简标志 (1=繁体, 0=标准)
  [2] 异体标志 (1=异体字)
  [1] 多音标志 (1=多音字)
  [0] 罕用标志 (1=罕用字)
```

## 六、部署建议

### 按场景选择版本

| 应用场景 | 推荐版本 | 理由 |
|---------|---------|------|
| 通用文本处理 | 8,105 版 | 覆盖 99.99% 日常用字，体积小 |
| 古籍数字化 | **97,686 版** | 必须覆盖扩展B+ |
| 字型检测 (GB/T 11460) | 8,105 版 | 检测用字集中 |
| 全文搜索引擎 | **97,686 版** | 完整覆盖 |
| 嵌入式/低内存 | 8,105 版 | 仅 531KB |
| AI 训练 | **97,686 版** | 完整语义空间 |

### 结构检测改进路线

```
当前 (9/16): 独体+左右+上下+5种包围
     ↓
短期 (10/16): 增加「品字结构」检测
     ↓
中期 (12/16): 增加「左中右+上中下」启发式规则
     ↓
远期 (16/16): 字符级形态分析 (CNN/字形识别)
```

## 八、结论

1. **两版编码方案完全兼容** — 共享 32-bit 位域、6 条自定义指令、哈希索引算法
2. **全量版修复前退化到 3 种结构类型，修复后恢复到 9 种** — 超过基础版的手动标注（6 种），代价是缺失了左中右/上中下/品字等字级分析类型
3. **扩展区两版均零占用** — 4bit 可安全用于元数据编码
4. **全量版推荐在需要完整 CJK 覆盖的场景使用**（古籍、搜索引擎、字体检测）
5. **基础版推荐在标准文本处理和嵌入式中使用**（体积小、结构标注精度高）
