# 贡献指南

感谢你对 CNBE-32 项目的兴趣！我们欢迎任何形式的贡献，包括但不限于：

- 报告 Bug 或提出改进建议（Issues）
- 提交代码修复或新功能（Pull Requests）
- 完善文档或翻译
- 在社区分享和推广

在开始贡献之前，请阅读并遵守我们的 [行为准则](./CODE_OF_CONDUCT.md)。

## 1. 报告问题 (Issues)

- 检查 **已有的 Issues**，避免重复报告。
- 使用 **清晰的标题** 描述问题，并提供重现步骤、预期结果和实际结果。
- 如果是数据问题（如编码错误），请提供受影响的字符和正确值。
- 如果是运行环境问题，请说明操作系统、Python 版本及依赖包版本。

## 2. 提交 Pull Request (PR)

我们采用 **GitHub Flow** 工作流：

1. **Fork** 本仓库到你的账户下。
2. **Clone** 你的 Fork 到本地。
3. 创建一个 **新的分支** 用于你的修改（请使用有意义的名称，如 `fix-radical-mapping` 或 `add-struct-support`）。
4. 进行修改并 **提交**（commit），提交信息请使用英文或中文，格式为 `<type>: <subject>`，例如 `fix: correct radical id for U+4E00`。
5. 确保你的代码符合项目的代码风格（见下文）。
6. 运行现有测试，确保没有破坏原有功能。
7. 将你的分支 **push** 到你的 Fork。
8. 在本仓库中 **创建 Pull Request**，描述你的修改内容和原因。

### PR 评审要求

- 所有 PR 至少需要一名维护者 Review。
- 如果 PR 涉及核心编码逻辑变更，必须提供新的测试用例证明正确性。
- 如果 PR 涉及新增数据（如新增部首映射），需注明数据来源。

## 3. 代码风格

- Python 代码遵循 [PEP 8](https://peps.python.org/pep-0008/) 风格。
- 使用 4 个空格缩进，禁止使用 Tab。
- 行长度限制为 120 字符（可适度放宽）。
- 函数和类应包含 docstring（使用 Google 或 NumPy 风格）。
- 提交前可运行 `black` 或 `autopep8` 自动格式化。

## 4. 测试

- 所有新功能或修复应包含相应的测试代码（放在 `tests/` 目录下）。
- 运行测试：
  ```bash
  python -m unittest discover tests

## 5. 文档贡献

文档位于 docs/ 目录，使用 Markdown 编写。

- 如果添加新章节，请在 README.md 的文档索引部分更新链接。

- 我们鼓励中英双语，中文为正式语言，英文供国际参考。

## 6. 数据贡献
- 本项目依赖 Unihan 数据，所有编码规则基于官方 Unicode 标准。

- 如果需要调整部首映射或结构类型，请附上 Unicode 官方来源的引用。

- 生成编码表时，请确保通过 cnbe_validator 脚本验证无重复和越界。

## 7. 法律与许可
- 所有贡献将被视为在 木兰宽松许可证 v2 下发布。

- 提交 PR 即表示您同意将您的贡献在相同的许可证下发布。

## 8. 联系方式

- 如有任何疑问，请通过 Issues 或发送邮件至 [你的邮箱地址] 联系维护者。

- 再次感谢你的贡献！

### 使用方法

1. 在项目根目录分别创建 `CODE_OF_CONDUCT.md` 和 `CONTRIBUTING.md` 文件。
2. 将上面的内容分别复制粘贴到对应文件中。
3. 将文件中 `[你的邮箱地址]` 替换为你自己的联系邮箱。
4. 提交并推送到 GitHub。

如果需要其他辅助文件（如 `PULL_REQUEST_TEMPLATE.md`、`ISSUE_TEMPLATE.md` 等），随时告诉我。
