# CNBE-32 Qwen3.5-0.8B LoRA 微调实验白皮书

**版本**: v1.0 | **日期**: 2026-07-08 | **状态**: 初步验证完成

---

## 一、实验概述

### 1.1 实验背景

CNBE-32（Chinese Native Binary Encoding）项目已完成从编码语义验证（v1-v6）到硬件实现（v7）、全中文操作系统（v8）、跨领域泛化（v9-v10.8）的完整技术栈闭环。前述实验均在**推理层面**验证了 CNBE 编码的语义可理解性与结构化先验优势，但尚未在**模型训练层面**检验编码知识是否能通过 LoRA 微调注入预训练语言模型。

本实验的目标是：**通过 LoRA 微调，将 CNBE-32 编码知识注入 Qwen3.5-0.8B 模型，使其能够理解并生成 CNBE-32 编码相关信息**。最终产出将部署至 Ollama 平台，用于 CNBE 编码方案的交互式演示。

### 1.2 实验目标

| 目标 | 说明 | 状态 |
|:----:|------|:----:|
| LoRA 微调管线建立 | 在 CUDA GPU 上完成从数据加载到 LoRA 权重合并的完整管线 | ✅ |
| 编码知识注入验证 | 模型能够理解 CNBE-32 编码概念并输出编码信息 | ✅ |
| 交互式演示部署 | 通过 Gradio Web 界面提供模型聊天演示 | ✅ |

### 1.3 实验设计思路

区别于传统的"预训练-指令微调-RLHF"长周期管线，本实验采用 **LoRA（Low-Rank Adaptation）** 方法，在预训练模型基础上高效注入结构化知识：

1. **基础模型选择**：Qwen3.5-0.8B（752M 参数），适合单 GPU 训练和推理
2. **训练目标**：CNBE-32 编码知识——部首编号、笔画数、结构类型的识别与生成
3. **训练规模**：500 步（约 22 分钟）的轻量微调，验证 LoRA 知识注入的可行性

## 三、训练数据

### 3.1 数据规模

| 数据集 | 样本数 | 文件大小 | 用途 |
|--------|:------:|:--------:|:----:|
| 完整数据集 | 178,034 | 718 MB | 全部生成数据 |
| 迷你数据集（本实验使用）| 1,000 | 472 KB | LoRA 微调 |

### 3.2 数据来源分布

迷你数据集涵盖 **19 个类别**：

| 类别组 | 样本数 | 说明 |
|--------|:------:|------|
| 通用语料 | 267 | AI 生成的通用中文文本 |
| 编码知识 | 281 | CNBE-32 编码方案直接知识 |
| 部首标注 | 169 | 汉字部首识别与编码 |
| 笔画标注 | 115 | 汉字笔画数识别 |
| 结构标注 | 71 | 汉字结构类型 |
| 古文经典 | 76 | 经典文献片段与注解 |
| 其他 | 21 | 综合标注、语义关联等 |

### 3.3 数据格式

每条样本包含 instruction（指令）、input（输入）、output（期望输出）、system（系统提示词）、source（来源）、category（类别）。

系统提示词：
```
你是一位精通 CNBE-32 中文原生二进制编码的 AI 助手。
CNBE-32 将汉字的部首、笔画、结构编码为 32 位整数。
高8位为部首，中5位为笔画，中4位为结构，低11位为字库索引。
```

训练采用 Chat Template 格式化，序列最大长度 256 tokens。

## 四、训练配置

### 4.1 LoRA 参数

| 参数 | 值 | 说明 |
|------|:---:|------|
| LoRA 秩 (r) | 32 | 中等秩，平衡容量与效率 |
| LoRA alpha | 64 | 缩放系数 alpha/r = 2 |
| LoRA dropout | 0.1 | 轻度正则化 |
| 目标模块 | 全部 7 个线性层（q/k/v/o/gate/up/down proj）|
| 可训练参数量 | ~33M（约 4.4%）|
| 适配器大小 | 51 MB |

### 4.2 训练超参数

| 参数 | 值 |
|------|:---:|
| 训练步数 | 500 |
| 轮数 | 3 |
| 有效批次大小 | 4（梯度累积）|
| 学习率 | 2e-4 |
| 优化器 | AdamW |
| 梯度裁剪 | 1.0 |

### 4.3 训练过程

| 步骤 | 损失值 | 累计耗时 |
|:----:|:------:|:--------:|
| 0 | - | 0s |
| 100 | 1.9045 | 266.9s |
| 200 | 0.8067 | 531.0s |
| 300 | 0.6314 | 800.1s |
| 400 | 1.7899 | 1058.3s |
| 500 | 0.7524 | 1322.1s |

**损失曲线**：前 300 步快速收敛（-67%），第 400 步反常回升（新增轮次遇到困难样本），第 500 步恢复至 0.7524。全程训练时间 **22 分钟**。

### 4.4 LoRA 权重合并

训练完成后将 LoRA 适配器（51 MB）合并至基础模型（1.4 GB），输出路径 `outputs/cnbe-merged/`。

## 五、模型部署

### 5.1 部署架构

由于 Qwen3.5 架构在 Ollama 0.31.1 中未得到支持，采用 Python 原生部署方案：

```
[merged model] -> [Transformers + CUDA] -> [FastAPI + Gradio]
                                             |
                                             +-> http://localhost:7860 (Gradio Web UI)
                                             +-> http://localhost:8000/v1/chat/completions (API)
```

### 5.2 推理性能

| 指标 | 值 |
|------|:---:|
| 模型加载时间 | ~1.7s |
| GPU 显存占用 | 1.5 GB / 8.6 GB (17%) |
| 完整响应时间 | ~97s（128 tokens）|

**性能瓶颈**：Qwen3.5 使用 20 层 linear_attention，需要 `flash-linear-attention` CUDA 扩展实现高效推理。该扩展在 Windows 上无预编译 wheel。在 Linux/WSL2 上 `pip install fla` 可提升 10-20 倍推理速度。

## 六、实验结果

### 6.1 模型能力评估

通过 Gradio 接口测试，微调后的模型能够：
- 理解 CNBE-32 编码概念（32 位布局、部首/笔画/结构字段）
- 输出具体汉字的编码信息（部首编号、笔画数、结构类型）
- 对比 CNBE 与传统编码方案的差异

**测试示例**：
```
Q: What is CNBE-32 encoding? Please explain the bit layout.
A: 汉字由部首和笔画数组成，结构为左右型。这符合标准CNBE-32编码的定义...
```

### 6.2 已知问题

| 问题 | 原因 | 解决方案 |
|:----:|------|----------|
| 输出重复 | 训练数据含增强标签 | 清洗数据，去除标签 |
| 推理速度慢 | flash-linear-attention 缺失 | Linux 环境运行 |
| 自我纠正循环 | 模型推理中尝试修正 | 降低 temperature |
| 编码格式差异 | 训练数据描述不一致 | 统一系统提示词 |

### 6.3 训练规模对比预估

| 维度 | 当前 (500 步) | 预期 (5000 步) | 提升 |
|:----:|:--------------:|:--------------:|:----:|
| 训练时间 | 22 分钟 | ~3.5 小时 | - |
| 损失值 | 0.7524 | 预计 <0.3 | 更优收敛 |
| 输出质量 | 有重复/修正 | 稳定可靠 | 显著提升 |

## 八、结论与展望

### 8.1 核心结论

1. **LoRA 知识注入可行**：仅 500 步（22 分钟）的轻量 LoRA 微调即可将 CNBE-32 编码知识注入 0.8B 小模型。
2. **模型能理解编码概念**：微调后模型能识别汉字部首、笔画数、结构类型并以 CNBE-32 框架输出。
3. **GPU 需求极低**：RTX 4060 Ti（8GB）全程承载训练和推理，显存峰值仅 1.5GB。
4. **Windows 有限制**：flash-linear-attention 在 Windows 上无预编译包，推理速度受限。

### 8.2 改进方向

| 优先级 | 方向 | 预期效果 |
|:------:|------|----------|
| 高 | Linux/WSL2 环境部署（pip install fla）| 推理速度提升 10-20 倍 |
| 高 | 扩大训练至 5000 步 | 输出质量显著提升 |
| 中 | 清洗训练数据 | 消除输出重复 |
| 中 | Linux 下 GGUF 转换 + Ollama | 端到端演示闭环 |

### 8.3 与 CNBE-32 全系列的关系

本实验首次将 CNBE-32 从**"推理层面的语义验证"**推进至**"训练层面的知识注入"**：

```
v1-v6:   编码语义验证          AI 理解编码
v7:      硬件实现              编码被 CPU/FPGA 执行
v8:      全中文 OS             编码支撑操作系统
v9-v10.8: 跨领域泛化            编码在多领域有效
本实验:  LoRA 微调              编码知识注入模型
```
